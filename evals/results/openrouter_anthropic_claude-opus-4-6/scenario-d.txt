## Critical

### 1. Signed long comparison corrupts sort order for ~33% of z=16 tiles

**Commit:** `1fc95a26` (support zoom 16)
**File:** `FeatureGroup.java:177` + `SortableFeature.java:10-11` + `LongMerger.java:90-91`

The `encodeKey()` bit layout puts the 33-bit tile ID in bits 63–31 via `tile << 31`. For z=16, tile IDs range from 1,431,655,765 to 5,726,623,060. When `tile_id >= 2^32` (4,294,967,296), bit 63 is set, making the encoded key **negative** as a signed long.

Both `SortableFeature.compareTo()` and `LongMerger.lessThan()` use **signed** comparison (`key < o.key` / `ak < bk`). This means ~33% of z=16 tiles (those with `tile_id >= 2^32`, corresponding to ~1.43 billion tiles at high x-coordinates) sort **before all z=0–z=15 tiles**.

This breaks the fundamental invariant that features are sorted by zoom level ascending, corrupting the output archive for any run using `maxzoom=16`.

**z=0–z=15 are unaffected** — the maximum z=15 tile_id is 1,431,655,764, and `1,431,655,764 << 31` fits safely in a positive signed long.

**Fix:** Either change the comparisons to use `Long.compareUnsigned()`, or reduce `TILE_ID_BITS` to 32 (and reclaim 1 bit from sortKey), or use a different encoding that keeps the key always positive.

---

### 2. `TiledGeometry.encode()` sign-extension corrupts bitmap for filled polygons at z=16

**Commit:** `1fc95a26` (support zoom 16)
**Files:** `TiledGeometry.java:302,808`

The static `encode()` method at line 808 returns `int`:
```java
static int encode(long maxTilesAtThisZoom, long x, long y) {
    return (int) (maxTilesAtThisZoom * x + y);
}
```

At z=16, `maxTilesAtThisZoom = 65536`. For x=32768: `65536 * 32768 = 2^31`, which overflows to `Integer.MIN_VALUE` (-2,147,483,648).

In `getCoveredTiles()` at line 302:
```java
long colStart = encode(maxTilesAtThisZoom, entry.getKey(), 0);
```
The `int` return is **sign-extended** to `long`, producing `-2147483648L` instead of `+2147483648L`.

`RoaringBitmap.addOffset(yRanges.bitmap(), colStart)` then applies a **negative** offset, corrupting the bitmap for all filled polygon columns with `x >= 32768` at z=16 (the entire eastern half of the tile grid). This means filled polygon rendering is wrong for half the world at z=16.

**Note:** Direct `bitmap.add(encode(...))` and `bitmap.contains(encode(...))` pairs work correctly because RoaringBitmap treats ints as unsigned 32-bit values consistently. The bug is specifically in the `addOffset` path used for filled ranges.

**Fix:** Cast with `Integer.toUnsignedLong(encode(...))` at line 302, or change `encode()` to return `long`.

## High

### 3. SSRF via `resolveUrl()` — protocol-relative and non-HTTP schemes accepted

**Commit:** `7dbef041` (STAC Overture download)
**File:** `OvertureStac.java:257-261`

```java
static String resolveUrl(String base, String href) {
    if (href.startsWith("http://") || href.startsWith("https://")) {
        return href;
    }
    return URI.create(base).resolve(href).toString();
}
```

This is called with `href` values parsed from untrusted STAC JSON responses. Two attack vectors:

**(a) Protocol-relative URLs redirect to arbitrary hosts.** An href of `//evil.com/malicious.json` passes the prefix check, and `URI.resolve("//evil.com/malicious.json")` produces `https://evil.com/malicious.json` (inheriting the base scheme, replacing the host). A compromised or malicious STAC catalog entry can redirect all subsequent traversal to an attacker-controlled server.

**(b) `file://` and other schemes via absolute URI.** An href of `file:///etc/passwd` doesn't start with `http://` or `https://`, so it falls through to `URI.resolve()`. Java's `URI.resolve()` with a different-scheme absolute URI returns that URI directly. `Downloader.openStream()` then calls `URI.toURL().openConnection()`, which supports `file://`, `jar://`, `ftp://` etc. A malicious catalog can read local files.

**Fix:** After resolution, validate that the resulting URL scheme is `https` (or `http`). Reject any URL whose host differs from the expected STAC/S3/Azure hosts.

### 4. `Collectors.toMap` crashes on duplicate URL keys

**Commit:** `7dbef041` (STAC Overture download)
**File:** `Planetiler.java:597`

```java
.collect(Collectors.toMap(Map.Entry::getKey, Map.Entry::getValue));
```

`Collectors.toMap` with no merge function throws `IllegalStateException` on duplicate keys. If two STAC items produce the same parquet URL (legitimate in STAC catalogs with overlapping items), this crashes the entire pipeline.

**Fix:** Use `Collectors.toMap(Map.Entry::getKey, Map.Entry::getValue, (a, b) -> a)`.

## Medium

### 5. Stale parquet files from previous Overture releases pollute input

**Commit:** `7dbef041` (STAC Overture download)
**File:** `Planetiler.java:595-602`

When a new Overture release comes out with different filenames, old `.parquet` files from the previous release remain in the download directory. The glob at line 602 (`*.parquet`) picks up **all** parquet files in the `theme=.../type=...` directory, including stale ones. There's no cleanup mechanism.

**Impact:** Tile output contains duplicate/outdated features from previous releases. Silent data corruption.

**Fix:** Delete or move old files when downloading a new release, or track expected filenames and ignore unexpected ones.

### 6. No response size limit on STAC JSON fetch — OOM vector

**Commit:** `7dbef041` (STAC Overture download)
**File:** `OvertureStac.java:269-270`


Reads the entire stream with no size cap. A malicious server could return a multi-gigabyte JSON payload, causing `OutOfMemoryError`.

### 7. Antimeridian bbox handling incorrect — false positives

**Commit:** `7dbef041` (STAC Overture download)
**File:** `OvertureStac.java:227-245`

Both `collectionExtentIntersects()` and `itemBboxIntersects()` use JTS `Envelope.intersects()`. JTS `Envelope` does not handle antimeridian wrapping. A bbox crossing 180°/-180° (e.g., Fiji: `minLon=177, maxLon=-178`) creates `Envelope(-178, 177, ...)` — nearly the entire longitude range — always returning `true`. This wastes bandwidth but doesn't lose data.

**Commit:** `7dbef041` (STAC Overture download)
**File:** `Planetiler.java:590`
