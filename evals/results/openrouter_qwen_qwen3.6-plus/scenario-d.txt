### High

1. **ExecutorService never shuts down on error path** (`Planetiler.java:128`, `Planetiler.java:988`)
   - `virtualThreadPoolExecutor` is only closed in the `try` block at line 988 after `stats.close()`. If `run()` throws before reaching that point (e.g., `PlanetilerException` from `checkDiskSpace`, `checkMemory`, or any stage failure), the executor is never shut down. Virtual threads will leak. This is especially problematic since `addOvertureSource` submits `CompletableFuture.supplyAsync` tasks to this executor.
   
2. **Race condition in `addOvertureSource` — `profile()` not yet available** (`Planetiler.java:581-606`)
   - `addOvertureSource` is called during the builder phase, but `profile()` is null until `setProfile()` is called. The `CompletableFuture` submitted at line 589 calls `stac.getParquetUrls(theme, type, config.bounds())` which is fine, but the later `download()` method at line 1118 checks `profile.caresAboutSource(toDownload.id)` — if the profile hasn't been set yet when `download()` is called, `profile()` will lazily invoke `profileProvider` which may itself depend on state not yet initialized.

3. **`encodeKey` bit layout overflow risk** (`FeatureGroup.java:176-179`)
   - The bit layout allocates 33 bits for tile ID. At zoom 16, the total tile count is `sum(4^z for z=0..16) = (4^17 - 1) / 3 ≈ 572,662,306`, which fits in 30 bits. However, the shift `tile << (64 - TILE_ID_BITS)` = `tile << 31` means the tile value occupies bits 63..31. If `tile` ever exceeds `2^33 - 1`, the upper bits would be silently truncated. The `assert z <= MAX_MAXZOOM` in `TileCoord` is an assertion that can be disabled at runtime (`-da`), allowing invalid tiles to corrupt the key encoding.

### Medium

4. **Unchecked cast in `VectorTile.java` Hilbert return** (`VectorTile.java:452`, `VectorTile.java:1075`)
   - `Hilbert.hilbertXYToIndex` now returns `long` (it always did, but was previously implicitly truncated to `int`). The explicit `(int)` cast at lines 452 and 1075 silently truncates the result. At zoom 15, the max Hilbert index is `4^15 - 1 = 1,073,741,823` which fits in a signed int, so this is safe for now, but if zoom 16 were ever used in these code paths, values up to `4^16 - 1 = 4,294,967,295` would overflow to negative integers.

5. **`Downloader.run()` — progress loggers built after futures start** (`Downloader.java:162-189`)
   - `CompletableFuture.allOf(...)` starts all downloads at line 163, but the progress loggers are constructed afterward at line 169-187. There's a window where downloads are running but no progress is being tracked. More critically, if a download completes before the loggers are set up, `item.metadata.get(10, TimeUnit.SECONDS)` at line 177 could throw a `TimeoutException` if the metadata future hasn't been completed yet — but this is unlikely since `downloadIfNecessary` completes metadata first.

6. **`OvertureStac` — no retry or error handling for HTTP fetches** (`OvertureStac.java:268-273`)
   - The `fetch` method wraps `Downloader.openStream` in a try-with-resources and throws `UncheckedIOException` on failure. There is no retry logic, unlike the `Downloader` itself which supports configurable retries. A transient network failure during STAC catalog traversal will fail the entire Overture source download with no recovery.

7. **`addOvertureSource` — `Glob.find()` may return empty list silently** (`Planetiler.java:602`)
   - The `Supplier<List<Path>>` passed to `addParquetSource` calls `Glob.of(downloadDir).resolve(...).find()`. If the download hasn't completed or files are placed in a different directory structure, this returns an empty list. The `ParquetReader.process()` will then process zero files with no warning, producing an empty tileset.

8. **`TileCoord` — `assert` used for validation that can be disabled** (`TileCoord.java:56`)
   - `assert z <= MAX_MAXZOOM` in the compact constructor is the only guard against invalid zoom levels. In production builds with assertions disabled, a zoom level > 16 would cause silent data corruption in the encoding/decoding logic.

### Low

9. **`FeatureGroup.SORT_KEY_BITS` reduced from 23 to 22** (`FeatureGroup.java:56`)
   - This is a deliberate change to accommodate the 33-bit tile ID, but it halves the sort key range from ~4M to ~2M. Profiles using sort keys near the old limits (`SORT_KEY_MAX` was ~4M, now ~2M) may silently overflow. No migration path or warning is provided.

10. **`ToDownload` record change breaks backward compat** (`Planetiler.java:1141-1145`)
    - The `ToDownload` record changed from `(String id, String url, Path path)` to `(String id, CompletableFuture<Map<String, Path>> urlToPath)` with a convenience constructor. Any external code or reflection-based tooling relying on the old record component names (`url`, `path`) will break.

11. **`ParksOverlay` example hardcodes `monaco` as default area** (`ParksOverlay.java:71`)
    - Minor: the example defaults to Monaco, which is fine for a demo but could confuse users who expect it to work with Overture data (which doesn't cover Monaco at the same granularity as OSM).
