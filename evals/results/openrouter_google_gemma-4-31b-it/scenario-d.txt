I have reviewed the last 10 commits. Most changes are dependency updates or support for zoom level 16, which required migrating `TileCoord` encoding from `int` to `long`.

### Findings

**1. Potential Integer Overflow in `TiledGeometry.encode`**
- **Severity:** High
- **Location:** `planetiler-core/src/main/java/com/onthegomap/planetiler/render/TiledGeometry.java:781`
- **Issue:** The `encode` method casts the result of `maxTilesAtThisZoom * x + y` to an `int`. With zoom 16, `maxTilesAtThisZoom` is $2^{16} = 65536$. For the maximum $x$ ($65535$) and $y$ ($65535$), the result is $\approx 4.29 \times 10^9$, which exceeds `Integer.MAX_VALUE` ($2.14 \times 10^9$), causing a wrap-around to negative values. This will likely lead to incorrect tile indexing or `ArrayIndexOutOfBoundsException` in `RoaringBitmap`.

**2. Unbounded Resource Consumption in `OvertureStac`**
- **Severity:** Medium
- **Location:** `planetiler-core/src/main/java/com/onthegomap/planetiler/util/OvertureStac.java`
- **Issue:** The `getParquetUrls` method fetches all STAC items for a collection in parallel using a virtual thread executor. While virtual threads are lightweight, there is no limit on the number of concurrent HTTP requests to the STAC API, which could lead to rate-limiting or socket exhaustion when processing large collections.

**3. Design Problem: Mixed Parallelism Strategies**
- **Severity:** Low
- **Location:** `planetiler-core/src/main/java/com/onthegomap/planetiler/Planetiler.java`
- **Issue:** The project is introducing `Executors.newVirtualThreadPerTaskExecutor()` for some tasks (Overture downloads) while continuing to use traditional `ExecutorService` or custom `WorkerPipeline` for others. This inconsistency makes resource monitoring and tuning more difficult.
